假设一个极简Transformer层, h=4, T=2 (两块GPU)

输入 X = [x₁, x₂, x₃, x₄] ∈ ℝ4 (一个token, h=4)

标准 TP 做法: 权重 W ∈ ℝ4×4 切成左右两半

GPU₀ 拿 Wleft ∈ ℝ4×2, GPU₁ 拿 Wright ∈ ℝ4×2

但 LayerNorm 需要全部4个元素算均值:

μ = x₁+x₂+x₃+x₄4

→ GPU₀ 和 GPU₁ 都必须存完整的 [x₁,x₂,x₃,x₄]

这就是标准 attention 中的冗余操作

缺点: 每次读完整X → 内存大, 浪费

Sequence Parallel 的做法:

假设 seq_len=4, 即4个token, 每个token的h=4

X ∈ ℝ4×4 = [tok₁, tok₂, tok₃, tok₄]

关键观察: LN 对每个 token 独立!

LN(tok₁) 只需 tok₁ 的 4 个元素, 不需要 tok₂,₃,₄

所以沿 seq 维切成两块:

GPU₀ 拿 块1 = [tok₁, tok₂] ∈ ℝ2×4

GPU₁ 拿 块2 = [tok₃, tok₄] ∈ ℝ2×4

先对各块按标准做法:

GPU₀: LN([tok₁,tok₂]) → 本地完成, 无需通信!

GPU₁: LN([tok₃,tok₄]) → 本地完成, 无需通信!

修正操作 — 拼回去做GEMM:

GEMM 需要完整 X → AllGather: 2块拼回4个token

GPU₀: [tok₁,tok₂] → 收到tok₃,tok₄ → 完整 [tok₁..₄]

GPU₁: [tok₃,tok₄] → 收到tok₁,tok₂ → 完整 [tok₁..₄]

做完 GEMM 得 Y ∈ ℝ4×4

ReduceScatter: Y 归约+切回

GPU₀ 得 Y块1 ∈ ℝ2×4, GPU₁ 得 Y块2 ∈ ℝ2×4

通信量验证:

无SP: AllReduce = 2N·T−1T = 2×16×12 = 16

有SP: AG + RS = 8×12 + 8×12 = 4+4 = 8+8 = 16 相等!

内存对比 — 这就是赚到的地方:

无SP, 每GPU需存的激活 (只看LN+Dropout, 共4处):

LN输入: 4×4 = 16

Dropout: 4×4 = 16

MLP-LN: 4×4 = 16

MLP-Drop: 4×4 = 16

共 64 个元素 ← 因为每块GPU都存完整seq

有SP, 每GPU只存自己那半:

LN输入: 2×4 = 8

Dropout: 2×4 = 8

MLP-LN: 2×4 = 8

MLP-Drop: 2×4 = 8

共 32 个元素 ← 省了一半!

即对块1修正: 64 → 32, 节省比 = 2×

其余激活 (QKV, fc1等) 本来就被TP切了, 不变

代入真实值 Llama-70B, h=8192, s=8192, T=8, 20层:

无SP 4处冗余: 4 × 8192² × 2B × 20 = 8.6 GB

有SP ÷8: 4 × 8192²/8 × 2B × 20 = 1.1 GB

净省 7.5 GB / GPU, H100有80GB → 可多跑更大batch

得: 通信量不变, 激活内存减半以上, 免费午餐
