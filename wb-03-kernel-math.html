阶段1: Reduce-Scatter, p−1 步

GPUi 发 chunk[(i + p − j) mod p], 大小 N/p

step 1..p−2: recv + copy + send

Pipeline 1F1B Bubble

p 个 stage, m 个 micro-batch

非 interleaved 1F1B:

stagei warmup = (p − i − 1) 次 forward

stage0: warmup = p−1,   stagep−1: warmup = 0

bubble = p − 1m

p=16, m=32 → 46.9%

Interleaved (v 个 virtual chunk):

warmup = (p − rank − 1) × 2 + (v−1) × group_size

bubble ≈ p − 1m · v

p=16, m=32, v=4 → 11.7%

代价: 通信 ×v, 且 m mod group_size 需 ≥ p

否则尾部 micro-batch 组产生依赖 bubble

MFU:

MFU = 6ΦDT · W · Fpeak ≈ ηbubble · ηcomm · ηrecomp · ηmem

12288 GPU 实测:

ηbubble=0.883, ηcomm=0.94, ηrecomp=0.95, ηmem=0.97

Π = 55.2%

TAOCP 视角批判

Knuth 五性质 — Online Softmax:

① 有穷: ⌈N/Bc⌉ 块, 有限循环 ✓

② 确定: 每步 max + exp2f + sum, 无歧义 ✓

③ 输入: Q, K, V ∈ ℝN×d

④ 输出: O = softmax(QKT/√d) · V

⑤ 有效: 每步有界计算量 ✓

用户角度批判:

1. exp2f 精度: s 截断为 float32 → 每次误差 ~2−23

N=128K → 累积 ~105×2−23 ≈ 0.012

长序列尾部 softmax 概率偏移 ~1% → 悄悄影响注意力模式

2. FP8 scale: amax=0 时 sf 保持旧值 → 若首次 amax=0

scale 初始化为 1, 则 448/1 过大 → 首batch溢出

3. ηbubble × ηcomm 假设独立

实际: 梯度通信可隐藏在 bubble 中 → 乘法低估 MFU

系统角度批判:

1. Ring vs Tree reduce顺序不同: (a+b)+c ≠ a+(b+c)

→ 拓扑切换 = 训练不可复现

2. ZeRO-3 + TP 不兼容: AllGather恢复参数后才能 TP

→ 通信量翻倍, 实际不被同时使用

3. Interleaved PP 要求 m mod group_size ≥ p

→ 否则尾部产生依赖 bubble, 静默性能退化
